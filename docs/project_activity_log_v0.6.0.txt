8.2  DECISIONS & ARCHITECTURE
--------------------------------------------------------------------------------

DECISION: next-actions data sources
  Options considered:
    A) Only query the alerts table
    B) Alerts + overdue review detection + at-risk goal analysis
  CHOSEN: Option B — three-source approach gives richer, more actionable output
  Implementation:
    - Alerts table: active (is_resolved=False) compliance alerts
    - Auto-detected: clients with last_review_date > 90 days or null
    - Goal analysis: GoalAssessment run for each goal; flag if feasibility < 40
    - De-duplication: skip auto-detected REVIEW_OVERDUE if already in alerts

DECISION: meeting-prep approach
  Options considered:
    A) Return structured data (portfolio + goals as JSON)
    B) AI-generated natural language brief via Claude
  CHOSEN: Option B — provides immediately usable content for RM; demonstrates
          Claude's synthesis capability; data already available in AI tools
  Implementation: claude.complete() (not streaming) with 7-section prompt;
                  max_tokens=1500 to keep brief concise

DECISION: financial-plan persistence
  Question: Where to save generated plans?
  Options:
    A) New financial_plans DB table
    B) Existing compliance_documents table (SUITABILITY_ATTESTATION type)
  CHOSEN: Option B — reuses existing schema; plans are suitability documents
          under SEBI IA Regulations (Clause 19 rationale); no migration needed
  Lookup: GET /api/financial-plan/{client_id} retrieves most recent
          SUITABILITY_ATTESTATION doc for that client

DECISION: financial-plan prompt structure
  Decided on 8 sections matching SEBI IA Regulations requirements:
    1. Executive Summary
    2. Portfolio Review (with XIRR vs benchmark)
    3. Goal Roadmap (with GoalAssessment data)
    4. Tax Optimisation Strategy (compare_tax_regimes())
    5. Retirement Projection
    6. Risk Assessment
    7. Recommended Action Plan (with timeframes)
    8. Important Disclosures (SEBI Clause 19 rationale)
  System prompt: Claude acts as SEBI-registered investment adviser

DECISION: require_rm_or_compliance guard
  Compliance officers should be able to VIEW meeting preps and next actions
  (for audit purposes), but only RMs can GENERATE financial plans.
  Implementation: GET endpoints use require_rm_or_compliance;
                  POST /generate uses require_rm only

DECISION: ClientDetail tab structure
  Added 4 tabs: Overview | Portfolio | Goals | Meeting Prep
  Meeting Prep tab uses useMutation (not useQuery) — brief generated on demand,
  not on page load (avoid unnecessary API calls + Claude costs)

8.3  ACTIONS PERFORMED
--------------------------------------------------------------------------------

BACKEND — backend/app/api/rm.py (full rewrite):

  GET /api/rm/next-actions:
    - Query all clients with selectinload(alerts) + selectinload(goals)
    - For each client:
      * Iterate active alerts → build action dict with recommended_action
      * Check last_review_date vs (today - 90 days) threshold → REVIEW_OVERDUE
      * Run GoalAssessment for each goal → flag if feasibility_score < 40
    - Sort by _PRIORITY_ORDER dict (critical=0, high=1, medium=2, low=3)
    - Strip internal sort key before returning
    - Returns: {total_actions, actions[], generated_at}

  GET /api/rm/meeting-prep/{client_id}:
    - Validates UUID, fetches Client with goals + alerts via selectinload
    - Fetches Portfolio + Holdings separately
    - Calls _build_meeting_context() → structured text with all client data
    - Calls claude.complete() with 7-section prompt
    - Returns: {client_id, client_name, brief, generated_at, context_summary}
    - Logs: client_id, output_tokens

  Helper _get_recommended_action(alert_type):
    - Dict mapping AlertType → human-readable recommended action string
    - 10 alert types covered + default fallback

  Helper _build_meeting_context(client, portfolio):
    - CLIENT section: name, age, risk profile, income, tax regime, KYC
    - PORTFOLIO section: AUM, gain%, XIRR vs benchmark, allocation by asset class
    - GOALS section: each goal with GoalAssessment (feasibility, status, shortfall)
    - ACTIVE ALERTS section: priority + message for each unresolved alert

  GET /api/rm/alerts/{client_id}:
    - Enriched with recommended_action field
    - UUID validation added
    - Changed guard from require_rm to require_rm_or_compliance

BACKEND — backend/app/api/financial_plan.py (full rewrite):

  POST /api/financial-plan/generate:
    - GeneratePlanRequest model: {client_id, advisor_notes, target_retirement_age,
      desired_monthly_income}
    - Fetches Client (goals + alerts), Portfolio (holdings)
    - Calls _build_plan_context() → comprehensive ~50-line context string including:
      * Client profile
      * Portfolio: AUM, XIRR, allocation, all holdings with gain%
      * Goals: each with GoalAssessment (feasibility, shortfall, SIP)
      * Tax: compare_tax_regimes() → recommended regime + savings
      * Alerts: all active
    - Calls claude.complete(max_tokens=3000, temperature=0.1)
    - Saves ComplianceDocument (SUITABILITY_ATTESTATION) to DB
    - Returns: {client_name, plan_id, plan, generated_at, generated_by, tokens_used}

  GET /api/financial-plan/{client_id}:
    - Queries ComplianceDocument WHERE client_id + doc_type=SUITABILITY_ATTESTATION
    - ORDER BY created_at DESC LIMIT 1 (most recent plan)
    - Returns plan text + metadata, or helpful message if no plan exists

  Helper _build_plan_context(client, portfolio, payload):
    - Structured with === section headers
    - Includes all client data + tax comparison + allocation + per-holding data
